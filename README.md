# Azure Document Intelligence — RAG over your documents

A retrieval-augmented document Q&A service: ingest documents, retrieve the most
relevant passages, and answer questions grounded in them with citations. Every
component has a **local default** and an **Azure backend**, so the exact same
code runs offline on a laptop or fully on Azure.

## Components and backends

| concern | local default | Azure backend |
| --- | --- | --- |
| retrieval | TF-IDF + cosine | **Azure AI Search** (vector) |
| answering | extractive (sentence selection) | **Azure OpenAI** (grounded chat) |
| document store | filesystem | **Azure Blob Storage** |
| feedback (NoSQL) | JSONL file | **Azure Cosmos DB** |

Switch any component by setting its `*_BACKEND` env var plus the relevant Azure
credentials (see `.env.example`). Mix and match freely — e.g. local retrieval
with Cosmos feedback.

## How it works

```
ingest ─► chunk ─► embed/index ─► vector store
ask    ─► retrieve top-k ─► answer with citations ─► log feedback (NoSQL)
```

In production the chunks are embedded by Azure OpenAI and indexed in Azure AI
Search (HNSW vector search); answers are generated by Azure OpenAI grounded
strictly in the retrieved chunks; conversations and thumbs-up/down land in
Cosmos DB to drive later improvement.

## Layout

```
src/
  chunking.py    # overlapping word-window chunks
  vectorstore.py # LocalVectorStore (TF-IDF) + AzureAISearchStore (vector)
  embeddings.py  # Azure OpenAI embeddings (Azure path)
  answer.py      # ExtractiveAnswerer (local) + AzureOpenAIAnswerer (grounded)
  docstore.py    # LocalDocStore + AzureBlobDocStore
  feedback.py    # LocalFeedbackStore (JSONL) + CosmosFeedbackStore (NoSQL)
  rag.py         # ingest / search / ask / feedback orchestration
  eval.py        # hit-rate@k + MRR
  serve.py       # FastAPI: /ingest /search /ask /feedback /health
data/sample_docs/, data/qrels.json
scripts/ingest_sample.py
infra/main.bicep             # Storage + AI Search + Azure OpenAI + Cosmos + Container App
.github/workflows/           # ci (tests + eval), cd (build + deploy)
tests/
```

## Run locally

```bash
pip install -r requirements.txt
PYTHONPATH=. python -m scripts.ingest_sample          # ingest corpus + retrieval eval
PYTHONPATH=. uvicorn src.serve:app --reload
# ask a question:
curl -s -X POST localhost:8000/ask -H "content-type: application/json" \
  -d '{"query":"How does Container Apps scale when there is no traffic?"}'
pytest -q
```

### Verify the Azure Blob path with Azurite

```bash
npx azurite --silent --skipApiVersionCheck
export DOC_BACKEND=azure_blob
export AZURE_STORAGE_CONNECTION_STRING="UseDevelopmentStorage=true"
PYTHONPATH=. python -m scripts.ingest_sample          # documents now stored in Blob
```

## Deploy to Azure (your subscription)

```bash
az login
az group create -n docintel-rg -l westeurope
az deployment group create -g docintel-rg -f infra/main.bicep \
  -p containerImage=<acr>.azurecr.io/docintel-api:v1
```

`infra/main.bicep` provisions Storage, Azure AI Search, Azure OpenAI, Cosmos DB,
Log Analytics, App Insights, and the Container Apps environment + app, wiring the
service to all four Azure backends. Or push to `main` and let the `cd` workflow
build + deploy (set the `AZURE_*` and `ACR_NAME` secrets first).

## What's verified here vs your step

Verified locally: chunking, retrieval and the retrieval evaluation, the
extractive answerer with citations, the feedback store, the FastAPI service, the
Blob document store against **Azurite**, the unit tests, and `bicep build` of the
infrastructure. The Azure AI Search, Azure OpenAI, and Cosmos data-plane paths
are real SDK code selected by config and provisioned by the Bicep; running them
live needs those Azure resources, which is the one step that requires your
subscription.
